# Retro 001: `gh auth status` маскировал сетевую проблему под invalid token

Связанный инцидент: [incident-001--gh-token.md](./incidents/incident-001--gh-token.md)

## Когда применять

Когда CLI-инструмент пишет про auth или токен, но рядом есть признаки, что проблема может быть в самой среде выполнения.

## Симптом

`gh auth status -t` в агентской сессии показывал invalid token, хотя пользователь в соседней SSH-сессии видел валидный login и scopes.

## Что произошло

Мы слишком быстро приняли текст ошибки `gh auth status` за прямое описание проблемы токена. Позже debug-вывод показал, что `gh` не мог выполнить даже DNS/API-запрос к `api.github.com`, поэтому сообщение про invalid token было следствием более низкоуровневого сбоя.

## Почему диагностика затянулась

- Смешались три независимых слоя: `git` credentials, `gh` auth state и сетевой доступ к GitHub API.
- Недостаточно рано была проверена среда выполнения агента как отдельная гипотеза.
- Успешная проверка в другой SSH-сессии слишком долго переносилась на агентскую сессию как будто они эквивалентны.
- Конфликтующие сигналы не были использованы сразу как повод проверить sandbox/network.

## Как диагностировать в следующий раз

1. Сразу разделить:
   - доступ `git` к remote;
   - auth-состояние конкретной тулзы;
   - сетевую доступность внешнего API из текущей runtime-сессии.
2. Если текст ошибки говорит про token/auth, но есть хотя бы один конфликтующий сигнал, запустить debug-режим и проверить фактический сетевой слой.
3. Не переносить результат команды из другой SSH-сессии на агент без явной проверки sandbox/network restrictions.
4. Для `gh` начинать с пары проверок:
   - `gh auth status -t`
   - `GH_DEBUG=api gh auth status -t`

## Вывод / правило

Если верхнеуровневая команда пишет про invalid token, это ещё не доказательство проблемы токена. Сначала нужно убедиться, что из текущей сессии вообще достижим внешний API, от которого зависит эта команда.
